# Explanation

Explanation pages describe why benchmatrix works the way it does. Use them for
architecture, policy, security posture, and maintenance context rather than
step-by-step instructions.

## Available explanations

* [Architecture](architecture.md), covering package boundaries and design
    constraints.
* [Performance model](performance.md), covering latency, throughput, and tail
    latency views.
* [Threat model](threat-model.md), covering supply-chain, benchmark input, and
    repository risks.
* [Dependency policy](dependency-policy.md), covering runtime and development
    dependency expectations.
* [License policy](license-policy.md), covering MIT-compatible project content
    and dependencies.
* [Lifecycle](lifecycle.md), covering branch, version, and support expectations.
* [Repository health](repository-health.md), covering maintenance signals and
    cadence.
* [Secrets](secrets.md), covering local checks and response steps.
* [AI-agent guidance](ai.md), covering operational expectations for automated
    contributors.
* [Deprecations](deprecations.md), covering public API migration expectations.
* [GitHub Actions security](github-actions-security.md), covering CI permissions
    and workflow controls.
* [Deployment environments](deployment-environments.md), covering package,
    documentation, and container outputs.
* [Repository settings](repository-settings.md), covering external setup
    expectations.
* [Publishing](publishing.md), covering release artifacts and publication
    checks.
* [First release](first-release.md), covering readiness checks before the first
    public release.

Use how-to guides and runbooks when the next action needs to be concrete.
